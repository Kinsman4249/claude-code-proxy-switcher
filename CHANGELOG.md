# Changelog

This file tracks real changes to this repository. Entries are grouped into numbered rounds of work, with each individual change getting its own running number rather than restarting at 1 per round.

### Initial local-mode proxy and switch (round one)

1. Added `litellm_config.yaml`, a LiteLLM proxy config with no cloud model entries and no Anthropic API key anywhere. It maps `claude-haiku-4-5`, `claude-sonnet-5`, and `claude-opus-4-8`, the model names Claude Code's main session and sub-agents actually send, all to the same local Ollama backend. This exists because Anthropic's April 2026 policy change blocking subscription OAuth tokens in third-party proxies ruled out a cloud fallback that draws from a Pro/Max subscription instead of billed API usage. Rather than accept surprise direct billing on a fallback, the config has no cloud path at all: if Ollama isn't serving, requests fail cleanly with a connection error instead of silently charging anything.
2. Added `local-model.Modelfile`, building a custom Ollama tag from Qwen3.5-9B at the UD-Q5_K_XL quantization (6.74 GB), chosen over Qwen2.5-Coder-7B (the original pick), Gemma 4, and DeepSeek-R1 distills after comparing tool-calling benchmarks at the same VRAM tier. DeepSeek's local distills are reasoning-focused and less reliable at strict tool-call schemas; Gemma is a generalist rather than a coding specialist; Qwen3.5-9B has a much longer native context window and stronger BFCL/tool-use scores. The model is text-only by omission rather than by picking a separate quant: Qwen3.5 loads vision only if a separate mmproj file is downloaded alongside it, so simply not downloading that file is the entire difference. Context window is set to 20K tokens as a deliberate tradeoff against KV cache headroom on a 7.7 GB usable VRAM card.
3. Added `litellm-ollama-box.service`, a host-side systemd `--user` unit that starts only the LiteLLM proxy inside a Distrobox container at login, without starting Ollama itself or loading any model. This keeps VRAM at zero until the user deliberately runs `ollama serve` inside the container. The unit runs entirely through `distrobox enter`, so it requires no `dnf`/`rpm-ostree` package layering on an immutable host, both Distrobox and systemd are already part of a stock install.
4. Added `distrobox-reminder.service`, a systemd `--user` unit that fires a `notify-send -u critical` desktop notification at login reminding the user to stop the container before a GPU-heavy task like gaming. Untested against every possible notification daemon; behavior (staying up until clicked vs. timing out) depends on the desktop's own notification server.
5. Added `claude-local-toggle.sh`, the actual on/off switch. It edits `~/.claude/settings.json`'s `env` block to add or remove the proxy override, since that file is read by both the Claude Code CLI and the VS Code/VSCodium extension. Backs up the pre-existing settings file once, on first use, so `off` can always return to a known-good state. Supports `on`, `off`, and `status` subcommands.
6. Added `claude-local-desktop-toggle.sh` and `claude-local-toggle.desktop`, a double-click desktop icon wrapping the toggle script. It reads the current state, flips it, and confirms the new state with a desktop notification, since a desktop icon launch has no terminal to print to.
7. Added `install.sh`, an interactive installer that runs every manual step above in order: places `litellm_config.yaml`, installs and enables both systemd units, optionally enables `loginctl` lingering, installs the toggle script and desktop shortcut with all paths and ports consistently patched across files, downloads the Qwen3.5-9B GGUF and builds the Ollama tag inside the container, and confirms LiteLLM itself is installed there. Every decision point is an interactive prompt that shows the previous run's answer as the default, saved to `~/.config/claude-local-setup.conf`, so a second run is mostly pressing Enter. Uses `set -uo pipefail` rather than `set -e` so one failed step reports a warning and continues instead of aborting the whole run partway through.

### Project scaffolding (round two)

8. Added a Code of Conduct based on Contributor Covenant 2.1, a Contributing guide covering bug reports, feature requests, PR submission, coding conventions (including a project-specific rule that no change may reintroduce an implicit cloud fallback without calling it out explicitly), and pre-1.0 versioning rules.
9. Added a Security policy scoped to this project's actual attack surface: the install and toggle scripts, the LiteLLM config, the systemd units, and specifically the requirement that local mode never requires or transmits an Anthropic API key.
10. Added generic bug report and feature request issue templates, and a pull request template, unmodified from the standard template set since neither needed project-specific placeholders filled in.
11. Added `.claude/settings.json` with Read-deny rules for build artifacts, lockfiles, `node_modules`, `dist`, and generated directories, copied from the standard template set.
12. Added this project's own tag-triggered release workflow (`.github/workflows/release.yml`), adapted from the text-only repo pattern: a single Ubuntu runner packages the tagged commit into `.tar.gz` and `.zip` source archives via `git archive` and attaches them to a GitHub Release with auto-generated notes. No build matrix, since there is nothing here to compile per platform.

### Installer fixes from first real run (round three)

13. Fixed `install.sh`'s container-existence check to match case-insensitively and resolve to the exact name `distrobox list` reports, rather than requiring an exact string match against whatever the user typed. This surfaced because a container-manager GUI (Kontainer) displays container names title-cased for readability even though the underlying Distrobox container name is lowercase, so a user copying the displayed name got a false "container not found" error against a container that was already running. The corrected name is written back to the saved config so later runs never hit this again, and a one-line note is printed the first time the correction happens so it isn't a silent behavior change.
14. Fixed the model-download and litellm-install steps failing with "pip: command not found" against a bare Fedora 41 Distrobox image, which does not ship pip by default the way some other base images do. Both steps now check for `python3 -m pip` first and install `python3-pip` via `dnf` if it's missing, and call pip unambiguously as `python3 -m pip` rather than assuming a bare `pip` binary exists on PATH.
15. Fixed both pip install steps then failing on permission errors (or succeeding but placing the resulting `hf` and `litellm` console scripts somewhere not on PATH) when run as a non-root user with `--break-system-packages`, since that flag does not grant write access to system site-packages by itself. Both installs now run under `sudo` inside the container, matching how the rest of the container's package management already works, so the resulting binaries land in a system-wide location already on PATH.
16. Fixed the model download itself failing with "Repo id must be in the form 'repo_name' or 'namespace/repo_name'" when using the `hf://` URI form of `hf download`. The container's installed `huggingface_hub` version (1.17.0) predates support for that syntax, even immediately after requesting an upgrade to it, since the upgrade did not actually take effect as expected in that environment. Switched to the older, more broadly compatible flag-based syntax (`hf download <repo> --include <filename> --local-dir <dir>`), which has been stable across a much wider range of installed versions and does not depend on the upgrade having succeeded.

### Model name mismatch found via live traffic (round four)

17. Fixed Claude Code's Haiku sub-agent calls failing with `Invalid model name passed in model=claude-haiku-4-5-20251001` even though the proxy was up, Ollama was serving, and `local-qwen` and `claude-haiku-4-5` both appeared correctly in `/v1/models`. Confirmed via the actual error text that Claude Code sends the full dated model string `claude-haiku-4-5-20251001`, not the short `claude-haiku-4-5` alias the config previously only matched. Added `claude-haiku-4-5-20251001` as its own `model_name` entry pointing at the same local backend, kept alongside the short alias rather than replacing it, since a future Claude Code release could plausibly use either form. `claude-sonnet-5` and `claude-opus-4-8` have not shown the same dated-string mismatch in testing so far, but the same fix pattern applies if they ever do: check the exact string in a 400 error or in the proxy's debug log, then add a matching `model_name` entry rather than assuming the short alias is what's actually sent.

### Ollama build path resolution fix (round five)

18. Fixed `ollama create` failing with `Error: 400 Bad Request: invalid model name` immediately after a successful model download. The Modelfile's `FROM ./Qwen3.5-9B-UD-Q5_K_XL.gguf` is a relative path, and `install.sh` was only doing `cd ~` before invoking `ollama create` against the Modelfile, which does not guarantee Ollama resolves the relative path the same way, especially since `hf download` can nest the downloaded file under a subdirectory depending on a given repo's internal layout rather than placing it flat in the target directory. When Ollama can't find a file at the path a `FROM` line specifies, it silently falls back to treating the value as a registry model name instead, which is where the misleading "invalid model name" text came from. `install.sh` now searches for the actual downloaded file under the container's home directory up to three levels deep, rewrites the Modelfile's `FROM` line to that file's absolute path in a temporary copy, and builds from the temporary copy instead of the original. `local-model.Modelfile` itself is left with the original relative path and a note that manual (non-install.sh) builds need to replace that line with wherever the file actually landed.

### Interactive quant and context-length selection (round six)

19. Fixed context truncation discovered on real Claude Code traffic: proxy logs showed `truncated = 1` at `n_tokens = 20479`, one token under the previous `num_ctx = 20480` default, meaning Claude Code's system prompt and tool schemas alone were filling the entire context window before the actual task content, which explained garbled or incomplete responses that had looked like a model malfunction rather than a configuration limit.
20. Replaced the single hardcoded quant/context combination with an interactive menu in `install.sh`: six curated Qwen3.5-9B quant options (Q4_K_M through Q6_K, with real file sizes and a one-line tradeoff note each) plus a custom-filename escape hatch, and a separate prompt for context length with the truncation finding above included as context for why the old default wasn't enough. Both answers are saved and shown as defaults on the next run, same as every other prompt.
21. Replaced the static `local-model.Modelfile` dependency in the build step with a Modelfile generated fresh on every run from the chosen quant's resolved absolute path and the chosen context length, so changing either setting never requires hand-editing a file, only answering the two prompts differently. The static `local-model.Modelfile` is kept in the repo as a reference for anyone building manually outside `install.sh`.
22. Added a download-skip check: before re-downloading, `install.sh` now searches the container's home directory for the requested filename and skips the download step entirely if it's already present, so switching only the context length for an already-downloaded quant does not re-pull several gigabytes unnecessarily.
23. Added a post-build verification step: after `ollama create` succeeds, `install.sh` loads the model once with a trivial prompt and prints `ollama ps` plus `nvidia-smi` memory usage (when available) immediately afterward, so the actual VRAM cost of a given quant/context combination is visible right away instead of being discovered later from a truncated response or an out-of-memory failure mid-session.

### Build-step dependency and health-check timing fixes (round seven)

24. Fixed `ollama create` failing with `could not connect to ollama server, run 'ollama serve' to start it` on a run where the model file was already downloaded but the Ollama server daemon had not been started yet. This is a direct consequence of `litellm-ollama-box.service` deliberately starting only the LiteLLM proxy and not `ollama serve`, so nothing loads into VRAM automatically at login, but it meant the build step required a manual `ollama serve` in another terminal first, which wasn't documented as a prerequisite for `install.sh` itself. The build step now checks whether the server is already running (`pgrep -f 'ollama serve'`) and starts it in the background with `nohup` if not, waiting up to 10 seconds for `ollama list` to succeed before proceeding. Starting the server process does not load any model weights; that still only happens on first actual inference, so this does not change the project's VRAM-at-idle behavior, it only removes a manual step that was previously required before the automated build could succeed.
25. Fixed the proxy health check firing a false-positive warning after a fixed 2-second sleep, which was not always enough time for LiteLLM to finish binding to its port right after `systemctl --user daemon-reload` and `enable --now`. The check now retries once per second for up to 10 seconds before reporting a warning, so a slow-starting proxy that comes up at, say, the 4-second mark no longer gets reported as down when it was simply still starting.

### Ollama server detection reliability (round eight)

26. Fixed the automatic `ollama serve` startup from round seven still failing with the same `could not connect to ollama server` error on a subsequent run. The previous check used `pgrep -f 'ollama serve'` to decide whether a server was already running, which can match a stale or crashed process that still appears in the process table but is no longer actually listening, causing the script to skip starting a working server and fail later with a confusing connection error at the `ollama create` step instead. Replaced the process-name check with an actual functional check (`ollama list` succeeding), extended the wait to 20 seconds, and added an explicit failure path: if the server still isn't responding after the wait, the script now prints the last 20 lines of the server's own log file and exits with a clear error, rather than silently continuing on to a build step that was always going to fail. This also surfaces genuine startup problems (a stale process still holding the port, a GPU driver issue, etc.) that were previously invisible.

### Ollama server startup moved to a visible terminal window (round nine)

27. Fixed `ollama serve` continuing to fail to stay up even after round eight's improved detection, root-caused to backgrounding it with `nohup`/`disown` inside a `distrobox enter -- bash -lc "..."` session in the first place: container runtimes can tear down every process spawned inside a given exec session once that session's own command exits, independent of `nohup`, so the server process was being killed the moment the download/build subshell finished, regardless of how well its presence was checked afterward. Replaced backgrounding entirely with opening `ollama serve` in its own terminal window (`konsole`, falling back to `gnome-terminal` or `xterm` if konsole isn't present), which keeps the process alive in a session the script never tears down, and doubles as the requested visibility into the server's own output for monitoring. The window stays open even if `ollama serve` exits or crashes, pausing on a keypress, so an error is visible instead of the window disappearing. The check for whether a server is already running, and the subsequent wait for a freshly opened one, both curl `http://localhost:11434/api/tags` directly from the host rather than entering the container, since Distrobox containers share the host's network namespace by default and the port is already reachable without an `enter`. If no supported terminal emulator is found, or the server still isn't reachable after 30 seconds, the download and build step is skipped with a clear message rather than proceeding into a step that would fail anyway; every other step in the script still runs normally.
